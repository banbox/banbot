# BanBot 后端业务流程与架构梳理

本文档基于当前仓库代码梳理后端整体架构、应用入口、核心业务流程和模块交互。文档只描述模块职责、业务流程和架构关系，不包含具体代码实现片段。

## 1. 总体定位

BanBot 是一个以量化交易为核心的 Go 后端系统，覆盖策略配置、交易对筛选、历史回测、参数优化、实盘交易、行情采集、数据存储、Web 控制台、通知与远程控制等完整链路。

系统的运行形态可以概括为四类：

- 命令行任务：初始化、回测、优化、下载/导入/导出/校验数据、工具类任务、实盘辅助任务。
- 开发 WebUI：策略管理、回测任务编排、数据工具、K 线/CSV/指标接口。
- 实盘交易进程：行情订阅、策略执行、订单管理、钱包与账户检查、通知、实盘监控 API。
- 行情采集进程：作为 spider 服务持续采集、落库并广播实时行情。

核心数据流是：

1. 配置和命令参数进入 `entry`。
2. 公共初始化由 `biz.SetupComs` 或 `biz.SetupComsExg` 完成。
3. 策略和交易对由 `goods`、`strat`、`biz` 共同生成运行任务。
4. 行情或第三方时序数据由 `data` 统一转换为 `orm.DataSeries`。
5. `biz.Trader` 将数据分发到策略，策略产出进出场请求。
6. `biz` 订单管理器按运行模式执行本地撮合或实盘交易所下单。
7. `orm` 持久化 K 线、时序数据、订单、任务和 UI 相关状态。
8. `web`、`rpc`、`live` 提供可观测、控制和通知能力。

## 2. 顶层架构分层

### 2.1 入口层

- `main`：进程主入口，只负责进入命令分发。
- `entry`：统一 CLI 分发、命令注册、参数绑定、信号处理、panic/退出钩子、性能分析开关。
- `web`：封装开发 WebUI 和实盘 API 两类 HTTP 入口。

### 2.2 配置与运行态层

- `config`：负责命令参数、配置文件、运行策略、账户、数据库、API Server、通知和 LLM 配置。
- `core`：保存运行模式、全局上下文、运行环境、缓存、退出钩子、常量与通用错误。
- `btime`：维护回测/实盘时间语义，使回测循环和实盘运行共享时间访问方式。

### 2.3 交易业务层

- `goods`：生成和过滤交易对。
- `strat`：策略注册、策略实例化、任务管理、数据订阅、策略回调和订单变化订阅。
- `biz`：公共启动初始化、策略任务刷新、交易执行协调、订单管理、钱包和工具任务。
- `live`：实盘交易主控制器、账户检查、运行时 cron、实盘平仓命令。
- `opt`：回测、参数优化、滚动优化、回测报告和工具型回放。

### 2.4 数据层

- `data`：历史数据 provider、实时 provider、feeder、spider、watcher、自定义时序数据源运行时。
- `orm`：K 线、交易所元数据、时序覆盖范围、第三方时序表、导入导出、数据库初始化。
- `orm/ormo`：交易订单和回测任务相关持久化。
- `orm/ormu`：UI 查询和开发 Web 任务相关持久化。
- `com`：实时价格缓存、bar 价格、定时任务通用能力。

### 2.5 接口与通知层

- `web/base`：K 线、CSV、指标、WebSocket 等通用 API。
- `web/dev`：开发 UI、策略文件、构建、回测任务、日志和数据工具 API。
- `web/live`：实盘 Dashboard API、登录鉴权、订单/账户/收益/配置/日志接口。
- `rpc`：Telegram、企业微信、Email 等通知与远程控制。
- `llm`：模型配置、模型实例缓存、并发控制、重试统计和 GLM/OpenAI 兼容调用。

## 3. 应用所有入口与业务流程

### 3.1 默认入口和命令分发

进程启动后进入 `entry.RunCmd`。它先安装退出和信号处理逻辑，再按参数决定运行方式：

- 无参数：启动开发 WebUI。
- 第一个参数是 `-h` 或 `--help`：打印命令帮助。
- 第一个参数是其他 flag：按 WebUI 参数处理。
- 第一个参数匹配命令组：进入对应子命令。
- 第一个参数匹配根命令：运行对应根命令。
- 未匹配：打印主帮助。

所有普通命令在执行前都会进入统一参数绑定流程，初始化 `config.CmdArgs`，设置运行模式、日志、性能分析选项，然后调用具体业务入口。

### 3.2 根命令

#### `trade`

实盘交易入口。主要流程：

1. 设置运行模式为实盘。
2. 执行公共初始化和交易所初始化。
3. 可选打开订单/数据转储文件。
4. 标记机器人运行状态和启动时间。
5. 创建 `live.CryptoTrader`。
6. 初始化实时数据源、订单管理、策略任务、Telegram 控制、Web API、账户检查和第三方数据源。
7. 启动实盘后台任务。
8. 进入实时行情循环。

#### `backtest`

历史回测入口。主要流程：

1. 设置运行模式为回测。
2. 初始化配置、交易所、数据库、交易对和策略上下文。
3. 计算默认输出目录。
4. 如果开启按策略分离回测，则逐个策略组执行。
5. 创建 `opt.BackTest`。
6. 初始化回测任务、交易对、策略任务和历史数据 provider。
7. 补齐所需历史数据和第三方时序数据。
8. 按时间推进历史数据，统一进入策略和订单流程。
9. 汇总并输出回测结果。

#### `spider`

行情采集入口。主要流程：

1. 设置实盘模式。
2. 初始化配置、数据库和基础组件。
3. 启动 spider websocket 服务端。
4. 清理未完成 K 线缓存。
5. 启动写库队列、监控和定时任务。
6. 根据客户端订阅创建 exchange miner。
7. 持续采集 K 线、成交、深度、价格等数据，落库后广播给订阅方。

#### `optimize`

超参数优化入口。它围绕回测引擎反复执行参数采样、回测评分和结果选择。采样策略支持多种优化器；每次采样会构造临时回测，计算评分后写入优化结果。

#### `bt_opt`

滚动优化回测入口。它按 review period 和 run period 划分时间轴，在每个窗口内用历史窗口优化策略参数，再把优化结果应用到后续运行窗口，继承钱包和历史订单状态，尽量模拟实盘中“用过去调参、用未来验证”的过程。

#### `web`

开发 WebUI 入口，等价于无参数默认启动。它初始化基础配置和数据库，注册通用 API 与开发 API，挂载前端静态资源，并监听 HTTP 端口。

#### `init`

初始化数据目录和默认配置，加载配置后完成基础目录与配置文件准备。

### 3.3 `data` 子命令组

- `data export`：从数据库导出 K 线和相关数据到 protobuf 文件。
- `data import`：从导出的 protobuf 数据目录导入数据库。

这组命令走公共初始化和交易所初始化，核心工作由 `orm` 的数据导入导出能力完成。

### 3.4 `kline` 子命令组

- `kline down`：按交易对、周期和时间范围从交易所下载 K 线。
- `kline load`：从 zip/csv 文件读取 K 线并写入数据库。
- `kline agg`：聚合大周期 K 线。
- `kline export`：从数据库导出 K 线 CSV。
- `kline purge`：按参数清理 K 线数据。
- `kline correct`：同步和校正不同周期 K 线。
- `kline verify`：验证 K 线表与覆盖范围元数据的一致性。
- `kline adj_calc`：重新计算复权因子。
- `kline adj_export`：导出复权因子。

这些命令的共同核心是 `orm` 的 K 线读写、覆盖范围、下载缺口判断和数据校验能力。

### 3.5 `tick` 子命令组

- `tick convert`：转换 tick 数据格式。
- `tick to_kline`：从 tick 构建 1m K 线。

该组命令属于离线数据加工，主要服务于高频历史数据回放和 K 线构建。

### 3.6 `tool` 子命令组

工具命令覆盖优化结果整理、回测模拟、滚动 picker 测试、日历加载、gRPC 数据服务、文件收益计算、相关性计算、资产报告合并、订单对账、策略列表、因子回测、回测结果重建和实盘 K 线一致性测试。

这些命令通常不会启动完整实盘主循环，而是复用 `biz`、`opt`、`data`、`orm`、`strat` 的局部能力完成专项任务。

### 3.7 `live` 子命令组

- `live down_order`：下载指定账户和交易所的历史订单。
- `live close_order`：按账户、交易对、策略等条件执行实盘平仓。

这组命令是实盘辅助入口，主要服务于外部运维和紧急处理。

### 3.8 Web 服务入口

#### 开发 WebUI

开发 WebUI 由 `web/dev` 启动。主要流程：

1. 解析 host、port、datadir、config、日志和时区参数。
2. 设置运行模式并执行 `biz.SetupComsExg`。
3. 在 Docker 场景下更新本地配置。
4. 收集已有回测结果并启动回测任务调度器。
5. 创建 Fiber 应用。
6. 注册 K 线、CSV、WebSocket 和开发 API。
7. 挂载前端静态资源。
8. 非 Docker 场景尝试打开浏览器。
9. 监听端口。

开发 API 覆盖策略树、策略文件、回测任务、构建、日志、可用策略、回测详情、回测订单、配置、数据工具、下载和报告管理。

#### 实盘 API

实盘 API 由 `live.CryptoTrader` 初始化期间调用 `web.StartApi` 启动。只有配置启用 API Server 时才会监听。主要流程：

1. 创建 Fiber 应用。
2. 按配置设置 CORS。
3. 注册通用 K 线、CSV、WebSocket API。
4. 注册 JWT 鉴权后的 `/api/bot` 业务 API。
5. 注册公开 API，如登录、ping、策略回调。
6. 挂载前端静态资源。
7. 后台 goroutine 监听端口。

实盘业务 API 覆盖版本、余额、钱包刷新、当日数量、统计、收益、任务交易对、交易所符号映射、订单、利润计算、平仓、交易所仓位关闭、延迟开仓、配置、策略任务、性能、交易下载、日志和机器人信息。

## 4. 公共初始化流程

多数后端入口最终都会调用 `biz.SetupComs` 或 `biz.SetupComsExg`。

`SetupComs` 负责：

1. 初始化命令参数。
2. 在实盘模式启用死锁检测。
3. 创建全局 context 和停止函数。
4. 初始化数据目录。
5. 加载配置。
6. 设置日志，实盘模式额外接入异常通知日志 core。
7. 初始化 core 缓存和全局基础设施。
8. 初始化邮件发送器。
9. 加载语言消息。
10. 初始化 utils、交易所、ORM、交易对过滤器。

`SetupComsExg` 在此基础上再初始化当前交易所市场元数据，使后续 K 线、交易和交易对流程能使用统一的 `ExSymbol` 映射。

这套初始化方式简化了入口开发，但也意味着大量模块依赖全局运行态，例如 `core`、`config`、`exg.Default`、`orm` 连接池、`goods` 过滤器和策略全局表。

## 5. 策略、交易对与任务流程

### 5.1 交易对筛选

`goods` 负责交易对来源和过滤。流程如下：

1. 从配置读取 pair filters。
2. 第一个 filter 必须能作为生产者生成候选交易对，除非配置了静态交易对。
3. 静态交易对会先按成交量和最小下单成本做修正。
4. 动态交易对从交易所市场列表和过滤器生产。
5. 后续过滤器按顺序执行，覆盖年龄、成交量、价格、涨跌幅、波动、相关性、黑名单、随机、偏移等维度。
6. 应用 offset 和 limit。
7. 写入 `core.Pairs` 和 `core.PairsMap`。
8. 策略配置中显式声明的交易对也会加入 `PairsMap`，确保运行期能被识别。

### 5.2 策略任务加载

`strat.LoadStratJobs` 是策略任务构建核心。主要流程：

1. 清空或重置任务相关全局状态。
2. 遍历 `config.RunPolicy`。
3. 通过策略工厂创建策略实例。
4. 读取策略级交易对过滤和 `OnSymbols` 输出。
5. 计算或读取每个交易对在不同周期上的评分。
6. 策略选择目标周期。
7. 按账号、策略、交易对、周期创建 `StratJob`。
8. 初始化每个任务的 `BarEnv`、DataHub、风控参数和订单限制。
9. 注册 WebSocket 订阅任务和第三方数据订阅。
10. 处理因交易对轮换而需要退出或继续跟踪的旧订单。

输出结果包括：

- 需要预热的交易对和周期。
- 账户维度的策略任务表。
- 侧输入数据任务表。
- 需要退出的旧订单。

### 5.3 数据订阅和侧输入

策略可以消费三类数据：

- 主 K 线：每个 `StratJob` 对应交易对和主运行周期。
- pair info：策略通过 `OnPairInfos` 声明的额外 K 线信息。
- 自定义时序数据：策略通过 `OnDataSubs` 声明的数据源、标的和周期。

`strat.CollectDataSubs` 会把策略声明转换为统一的 `DataSub`，后续由 `data.SeriesRuntime` 负责补齐历史、建表、订阅和激活。

### 5.4 策略执行

无论回测还是实盘，最终都进入 `biz.Trader.FeedDataSeries`。

对于 K 线数据：

1. 更新价格缓存和时间上下文。
2. 找到当前交易对和周期关联的策略任务。
3. 更新任务的 BarEnv 和订单状态。
4. 执行策略的 `OnData` 或 `OnBar`。
5. 执行自定义退出检查。
6. 将策略产生的进出场请求交给订单管理器。

对于非 K 线自定义时序数据：

1. 按 source、sid、timeframe 写入策略 DataHub。
2. 分发给声明了该订阅的任务。
3. 调用策略的 `OnData`。
4. 若策略产生订单请求，仍回到统一订单管理流程。

### 5.5 批量策略回调

策略可以把多个交易对或侧输入聚合到批量窗口中处理。`AddBatchJob` 记录待执行任务，`TryFireBatches` 在合适时间触发批量回调，并在回调后统一处理订单请求。

该设计适合横截面策略，但增加了时间窗口、并发和延迟执行复杂度。

## 6. 回测与优化流程

### 6.1 回测主流程

`opt.BackTest` 复用交易主链路，但用历史 provider 和本地订单管理器替代实时 provider 与实盘下单。流程如下：

1. 设置回测时间。
2. 初始化本地钱包和订单任务库。
3. 初始化交易所符号上市时间。
4. 刷新交易对和策略任务。
5. 补齐第三方时序数据。
6. 历史 provider 根据任务订阅 warmup 和正式区间。
7. 如交易所支持且历史数据缺失，按覆盖范围自动下载缺口。
8. 按时间顺序推进每个 feeder 的下一根数据。
9. 每根数据进入 `Trader.FeedDataSeries`。
10. 本地订单管理器按 K 线撮合挂单、止损、止盈和退出。
11. 结束后退出未完成任务，统计订单、收益、回撤、胜率和图表。

### 6.2 回测中的订单生命周期

回测订单由本地订单管理器处理。策略生成入场或出场请求后，订单管理器检查风控和数量限制，创建订单对象，并在后续 K 线内模拟成交、盈亏和钱包变化。订单变更仍会触发策略的订单回调，因此回测和实盘在策略接口上保持一致。

### 6.3 参数优化

超参优化以回测评分为目标函数。流程如下：

1. 解析策略可优化参数。
2. 按采样器生成参数组合。
3. 临时应用参数并执行内部回测。
4. 计算得分。
5. 记录结果并选择最优配置。

优化任务可以按策略、方向或交易对拆分，最终输出可复用的策略配置。

### 6.4 滚动优化

滚动优化把时间轴拆成“回看窗口”和“运行窗口”。每轮先在回看窗口优化，再在后续运行窗口回测，并继承上一轮钱包和历史订单状态。该流程用于降低未来函数风险，更接近实盘参数更新过程。

## 7. 实盘交易流程

### 7.1 CryptoTrader 启动流程

实盘主控制器 `CryptoTrader` 负责把实时数据、策略、订单和接口组合起来。启动流程如下：

1. 创建 `LiveProvider`，连接 spider。
2. 注入策略 pair 更新 hooks，使策略刷新时能触发行情订阅和订单退出。
3. 初始化任务库。
4. 初始化 Telegram 订单管理注入。
5. 启动实盘 Web API。
6. 在真实环境做账户检查。
7. 初始化订单管理器，真实交易走 live order manager，dry run 或特殊场景走本地 live order manager。
8. 加载交易对和策略任务。
9. 注册 WebSocket 取消订阅钩子。
10. 同步第三方时序数据。
11. 启动后台 runtime jobs。
12. 进入 `LiveProvider.LoopMain` 等待 spider 推送。

### 7.2 实时行情到策略

实时行情链路如下：

1. `data.RunSpider` 启动 websocket server 和 exchange miner。
2. miner 订阅或轮询交易所 K 线、成交、深度、价格。
3. 完成 K 线先写入数据库，再广播给 watcher。
4. `SeriesWatcher` 接收消息，按订阅任务筛选。
5. `LiveProvider` 将 K 线转换为 `DataSeries`。
6. `SeriesFeeder` 处理多周期聚合、未完成 bar、缺失修复和回调触发。
7. 回调进入 `CryptoTrader.FeedDataSeries`。
8. `Trader` 分发给策略并触发订单处理。

成交和深度类 WebSocket 数据不走主 K 线聚合，而是直接路由到策略的 `OnWsTrades`、`OnWsDepth`、`OnWsData` 或 `OnWsKline` 回调。

### 7.3 实盘订单生命周期

实盘订单链路如下：

1. 策略产生入场或出场请求。
2. `OrderMgr.ProcessOrders` 做统一风控、数量、方向、账户和策略限制检查。
3. 订单对象写入内存和持久化状态。
4. live order manager 把订单放入异步队列。
5. 队列消费者调用交易所接口提交入场、退出、止损、触发单等请求。
6. 交易所成交通过私有 WebSocket 或轮询回补进入系统。
7. 系统更新成交数量、均价、手续费、状态和钱包占用。
8. 订单完成后写库、触发通知、清理触发单并更新策略回调。
9. 周期性任务持续对账本地订单、交易所 open orders、仓位和钱包余额。

启动恢复时，live order manager 会同步数据库订单和交易所订单，尽量恢复进程重启前的未完成状态。

### 7.4 钱包、账户检查和风控

实盘启动会执行账户权限、余额、持仓模式等前置检查。运行中钱包模块维护可用、冻结、已实现盈亏和持仓估值。价格缓存来自 bar、成交、深度或主动 ticker 刷新，供订单风控、钱包估值和展示使用。

后台 cron 负责余额同步、订单对账、K 线超时检测、全局止损、杠杆监控、触发单检查、策略输出落盘等任务。

### 7.5 远程控制

远程控制入口包括：

- Web 实盘 API：订单查询、平仓、交易所仓位关闭、延迟开仓、余额刷新、收益统计、日志查询。
- Telegram：订单列表、平仓、状态、钱包、禁用/启用交易、账户切换、菜单和回调按钮。
- CLI live 子命令：下载交易所订单、按条件平仓。

这些入口最终会调用 `biz` 的订单和钱包能力，但有些直接交易所平仓路径可能绕过部分策略语义，因此需要依赖后续对账恢复一致性。

## 8. 数据、行情与自定义时序

### 8.1 Provider 和 Feeder 抽象

`data.Provider` 管理多个交易对 holder。每个 holder 是一个 feeder，对应一个交易对和多个周期。

provider 的核心职责：

- 订阅或取消交易对和周期。
- 为新增周期执行 warmup。
- 维护 dirty 状态。
- 在历史模式按时间推进，在实盘模式接收 watcher 推送。

feeder 的核心职责：

- 维护每个周期的下一根数据时间。
- 处理 warmup 数据。
- 从最小周期聚合到目标周期。
- 处理未完成 bar 和等待 bar。
- 触发统一 `DataSeries` 回调。

### 8.2 历史数据 provider

`HistProvider` 用于回测。它从数据库读取历史 K 线，必要时调用交易所补下载缺失区间，并按时间顺序推进每个 feeder。高频成交数据可通过 `WsDataLoader` 下载并缓存为本地二进制文件，再供回测 trade feeder 使用。

### 8.3 实时数据 provider

`LiveProvider` 用于实盘。它连接 `SeriesWatcher`，向 spider 发送 watch/unwatch 消息，并在收到 K 线、成交或深度时分发给 feeder 或策略 WebSocket 回调。

当策略订阅新增交易对时：

1. provider 创建或更新 feeder。
2. 对新周期做 warmup。
3. 根据周期决定订阅 1m、hour 或指定周期。
4. 对策略声明的 trade/depth/kline WebSocket 订阅发送 watch 请求。
5. 对移除的交易对发送 unwatch。

### 8.4 Spider 采集服务

Spider 是独立的实时行情服务。它维护 websocket server，按订阅启动 exchange miner。miner 可以通过交易所 websocket 接收实时数据，也可以通过定时 REST polling 兜底抓取 K 线。

K 线落库流程强调“先写库、后广播”，使 live provider 收到的数据通常已经在数据库中可查询。写库任务通过队列削峰，失败时记录日志并重试或告警。

### 8.5 自定义时序数据源

自定义数据通过 `data.DataSource` 接入。数据源需要提供：

- `SeriesInfo`：描述表名、时间列、sid 列、end 列和字段。
- 历史拉取能力：用于回测和启动补齐。
- 可选实时订阅能力：用于实盘运行时推送。

策略通过 `OnDataSubs` 声明订阅，运行时会：

1. 收集所有策略任务的订阅。
2. 计算 warmup 起点和结束时间。
3. 建表并查找缺口。
4. 对缺口调用数据源历史拉取。
5. 写入时序仓储并更新覆盖范围。
6. 激活实时订阅。
7. 数据到达后通过 `DataSink.Emit` 转成 `DataSeries`，进入 `Trader.FeedDataSeries`。

因此，自定义数据和 K 线最终共享同一策略事件模型。

## 9. 存储架构

### 9.1 数据库分工

当前存储分为三类：

- K 线、交易所元数据、日历、复权因子、覆盖范围、自定义时序数据：使用 PGWire 后端，支持 QuestDB 和 Timescale/PostgreSQL 分支。
- 交易订单、回测任务、交易流水：使用独立 SQLite 文件，由 `orm/ormo` 管理。
- UI 相关任务和可变状态：使用 `banpub.db` SQLite 文件，由 `orm/ormu` 和部分 `orm` 公共能力管理。

`orm/readme.md` 明确约束：`ormo`、`ormu` 依赖 `orm`，不能反向依赖，以避免依赖环。

### 9.2 QuestDB 与 Timescale/PostgreSQL

`orm.Setup` 会建立 PGWire 连接池，按配置或端口识别后端类型：

- QuestDB：默认端口 8812，禁用 pgx statement cache，并运行 QuestDB migration。
- Timescale/PostgreSQL：默认端口 5432，运行 PostgreSQL schema 和 migration。

QuestDB 支持自动安装和启动，并根据配置调整内存参数。PostgreSQL/Timescale 路径保留标准连接池语义。

### 9.3 K 线和覆盖范围

K 线表按周期拆分，例如 1m、5m、1h、1d 等。系统使用 `ExSymbol` 的 sid 统一识别交易所、市场、真实交易所和 symbol。

覆盖范围由 sranges 抽象表达，记录某个 sid、表、周期下哪些毫秒区间已经覆盖，以及该区间是否有数据。该模型支持不连续覆盖和无数据空洞，下载时可以精确计算缺失区间，避免重复下载或误判完整。

在 QuestDB 下，sranges 使用 append-only 和软删除配合 LATEST BY；在 PostgreSQL 下，sranges 可以使用事务、更新和唯一约束表达同样语义。

### 9.4 QuestDB WAL 可见性

QuestDB WAL 表存在写入成功但立即读取不可见的问题。项目已有专门的可见性等待逻辑，用于 exsymbol、calendar、K 线窗口、K 线范围、series 写入和表重写校验。

架构上应把 QuestDB 视为最终一致读模型：任何会在写入后立刻做判断、删除、rename 或清理恢复状态的流程，都必须先等待目标行、范围或计数可见。

### 9.5 SeriesRepo 与自定义时序

`SeriesRepo` 为自定义时序数据提供统一仓储能力：

- 建表。
- 批量插入。
- 查询范围。
- 删除或重写范围。
- 更新覆盖范围。
- 获取已存在范围。

QuestDB 和 PostgreSQL 在时间列、upsert、删除方式、JSON 字段和可见性等待上有差异，但上层数据源和策略订阅不需要直接关心这些差异。

## 10. Web、通知与 LLM 辅助模块

### 10.1 Web API 分层

`web/base` 提供开发和实盘共享 API：

- K 线 symbol 查询、历史 K 线、指标列表和指标计算。
- CSV 上传、列表和读取。
- OHLCV WebSocket。
- 错误处理和基础响应封装。

`web/dev` 面向策略开发和回测编排：

- 策略文件树、文本读取和保存。
- 新建策略。
- 构建运行环境。
- 启动回测子进程并跟踪日志。
- 读取回测结果、订单、配置、HTML 报告。
- 数据工具、下载和报告管理。

`web/live` 面向实盘监控和控制：

- 登录、JWT 鉴权、ping 和策略回调。
- 账户、余额、订单、收益、配置、性能和日志查询。
- 平仓、关闭交易所仓位、延迟开仓、下载交易记录。

### 10.2 通知与远控

`rpc` 的通知总线按配置初始化通道。消息由业务传入统一 payload，再按 webhook 模板渲染，并投递到启用的 channel。

支持通道包括：

- Telegram：除了通知，还提供订单查询、平仓、钱包、状态、交易开关和账户切换。
- 企业微信：按 access token 发送文本消息，带失败重试。
- Email：通过 SMTP 发信，支持批量合并。

异常日志通知通过 zap core 接入，带缓存和限流，避免同类异常短时间内刷屏。

### 10.3 LLM 模块

`llm` 是独立辅助能力，不直接参与交易主循环。它负责：

- 解析模型配置和继承关系。
- 环境变量替换。
- 创建 OpenAI 兼容或 GLM 模型客户端。
- 按模型做实例缓存。
- 控制并发、超时和失败禁用。
- 记录成功/失败和等待耗时统计。

该模块适合被后续策略生成、分析或开发工具调用，但当前架构上是边缘模块。

## 11. 核心交互时序

### 11.1 实盘主时序

1. `banbot trade`。
2. `entry` 解析参数并调用 `RunTrade`。
3. `biz.SetupComsExg` 初始化配置、日志、交易所、数据库和交易对过滤器。
4. `CryptoTrader` 创建 live provider。
5. `goods` 生成交易对。
6. `strat` 加载策略任务和数据订阅。
7. `data.SeriesRuntime` 补齐第三方时序并激活订阅。
8. `LiveProvider` 向 spider 订阅行情。
9. spider 推送 K 线、成交、深度。
10. provider/feeder 转成 `DataSeries`。
11. `Trader` 调用策略。
12. 策略产生订单请求。
13. live order manager 异步提交交易所。
14. 成交回报更新订单、钱包和通知。
15. Web/API/Telegram 可查询和控制。

### 11.2 回测主时序

1. `banbot backtest`。
2. `entry` 解析参数并调用 `RunBackTest`。
3. `biz.SetupComsExg` 初始化环境。
4. `opt.NewBackTest` 创建回测引擎和历史 provider。
5. `goods`、`strat` 构建交易对和策略任务。
6. `data.SeriesRuntime` 补齐第三方时序。
7. `HistProvider` 补齐和加载历史 K 线。
8. provider 按时间推进 feeder。
9. `Trader` 调用策略。
10. 本地订单管理器撮合订单并更新钱包。
11. 回测结束后汇总报告。

### 11.3 自定义时序数据时序

1. 外部模块注册 `DataSource`。
2. 策略通过 `OnDataSubs` 声明订阅。
3. 策略任务加载时收集订阅。
4. `SeriesRuntime` 规划 warmup 区间。
5. 仓储检查缺口并调用 `FetchHistory`。
6. 数据写入 series 表并更新覆盖范围。
7. 实盘模式激活 `SubscribeLive`。
8. 数据源通过 sink emit。
9. `Trader` 把数据写入 DataHub 并触发策略 `OnData`。

## 12. 架构分析与不够优雅的地方

### 12.3 策略任务和订单流程耦合较深

策略任务、订单管理、钱包、DataHub、pair rotation、订单回调之间交互紧密。好处是策略接口能力强；坏处是任务增删、订单恢复、批量回调和实盘异步成交叠加时理解成本高。

建议增加一份运行期状态机文档，并把订单状态转换、策略回调触发点和钱包变更点显式建模，配套状态迁移测试。

### 12.4 实盘异步一致性需要更强可观测性

实盘订单经历异步队列、交易所提交、私有成交回报、轮询回补、本地数据库更新和对账。该模型天然是最终一致的，短时间内 Web/API/Telegram 查询到 pending 或中间状态是正常现象。

建议对外部控制接口返回更明确的状态语义，例如 accepted、submitted、confirmed、filled、reconciled，并在 UI 中展示订单从请求到交易所确认的状态轨迹。

### 12.5 行情完整性判断逻辑复杂

实时 K 线在 watcher、provider、feeder 中都有未完成 bar、等待 bar、缺失修复、多周期聚合和 fallback 逻辑。该设计适应性强，但边界复杂。

建议把 bar 完成判定和多周期聚合抽成可独立测试的状态机，增加重连、乱序、延迟、缺失、1m/5m/1h 混合周期的测试矩阵。

### 12.6 存储后端差异被较好封装，但迁移复杂度高

`orm` 已经通过 `IsQuestDB`、sranges、SeriesRepo 等抽象封装了 QuestDB 和 PostgreSQL 差异。但两类后端在时间字段、可见性、删除、upsert、JSON、DDL 上仍存在大量分支。

建议将后端能力进一步接口化，例如 KLineStore、SRangeStore、SeriesStore、MetaStore，让上层业务不直接判断 `IsQuestDB`。同时保留 QuestDB read-after-write 的回归测试作为强约束。

### 12.7 Web 开发 API 启动外部命令需隔离

开发 Web 会触发构建、回测子进程、日志读取和文件操作。这些能力适合本地开发，但如果暴露到非本地环境，权限边界较大。

建议明确 dev server 只用于本地或受信环境，生产实盘只启用 live API；对构建和文件操作增加路径白名单、任务沙箱和审计日志。


## 13. 优化建议优先级

高优先级：

- 增强实盘订单状态可观测性和外部控制幂等性。
- 把 QuestDB 写后读可见性规则继续固化到所有写后判断、表交换和恢复状态清理流程。
- 为实时 K 线完成判定、多周期聚合、重连缺失补齐增加专门测试。
- 为远控入口统一命令、审计、限流和权限模型。

中优先级：

- 拆分 `SetupComs` 初始化能力，降低入口不必要依赖。
- 减少策略/回测/实盘共享全局状态，逐步引入运行上下文。
- 抽象存储后端接口，减少业务代码中的后端分支。
- 为 DataSource 热加载和多实例部署补充跨进程治理。

低优先级：

- 清理小型重复逻辑和冗余字段赋值。
- 补充面向架构的包级 README。
- 将开发 Web 的命令执行和文件操作边界文档化。
